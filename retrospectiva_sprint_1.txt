# Reporte Retrospectivo: Sprint 1 - Módulo de Autenticación y Registro (Report ARG)

## 1. Visión General y Arquitectura Frontend
En este primer Sprint, nuestro equipo construyó desde cero toda la arquitectura frontend para el sistema de autenticación de **Report ARG**. 

### Estructura del Frontend
El proyecto está construido sobre **Next.js (App Router)**. Hemos adoptado una arquitectura basada en componentes modulares y diseño atómico para asegurar un código escalable y fácil de mantener:
- **`src/app/`**: Manejo de rutas principales (`/login`, `/register/citizen`, `/register/institution`, `/verify`). Utilizamos *Server Components* donde es posible para optimizar el rendimiento y SEO, delegando la interactividad a los *Client Components*.
- **`src/components/`**: Contenedores principales lógicos de los formularios (`Login.jsx`, `CitizenRegister.jsx`, `InstitutionRegister.jsx`). Aquí reside el estado y el uso de *hooks*.
- **`src/components/auth/`**: Componentes puramente visuales extraídos para mantener el *Clean Code*. Por ejemplo, `AuthSplitLayout.jsx` actúa como un envoltorio global para la pantalla dividida, y los pasos individuales como `CitizenDataStep.jsx` renderizan la interfaz.
- **`src/components/ui/`**: Elementos de interfaz reutilizables compartidos en toda la app, como el indicador de progreso `StepIndicator.jsx`.
- **`src/utils/schemas.js`**: Centralización de esquemas de validación de datos usando **Zod**, garantizando un tipado fuerte y seguro antes de enviar cualquier carga útil (*payload*) al backend.
- **Estilos (`src/styles/` & Tailwind)**: Combinación de clases utilitarias de Tailwind CSS para diseño rápido en línea, complementado con CSS puro (`login.css`, `globals.css`) para definir variables corporativas, botones estándar (`.btn-primary`) y maquetaciones flexbox complejas.

## 2. División de Tareas e Implementaciones

El trabajo se dividió estratégicamente entre los miembros del equipo basándonos en las Historias de Usuario (HU). A continuación, se detalla el trabajo y la implementación técnica de cada uno.

### 👩‍💻 Julieta (~35 hs | SP: 10)
**Historias:** HU-01 (Selección de cuenta), HU-02 (Registro de ciudadano), HU-04 (Validaciones).

* **Implementaciones realizadas:** 
  - Desarrolló la pantalla de enrutamiento inicial (`AccountTypeSelector.jsx`) para clasificar a los usuarios de manera visual y clara.
  - Maquetó y programó el formulario de registro ciudadano. Para mejorar la experiencia del usuario y evitar una sobrecarga visual, estructuró el formulario bajo un patrón **"Multi-step Wizard"** de 3 pasos (Cuenta -> Datos -> Ubicación), controlando la navegación mediante variables de estado de React.
  - Llevó a cabo la integración de `react-hook-form` con los resolutores de `Zod`. Creó el esquema `citizenRegisterSchema`, el cual asegura validaciones asíncronas estrictas entre pasos (por ejemplo, validando nombre y contraseña antes de habilitar el avance a la pestaña de ubicación).

### 👩‍💻 Valentina (~38 hs | SP: 8)
**Historias:** HU-03 (Registro de institución), HU-05 (Confirmación por email), HU-06 (Verificación de instituciones).

* **Implementaciones realizadas:** 
  - Construyó el registro para entidades en `InstitutionRegister.jsx`, adaptando el *Wizard* para solicitar datos corporativos obligatorios (CUIT, Tipo de Institución) y exigiendo la "Dirección Exacta" en la capa de geolocalización.
  - Desarrolló desde cero la pantalla de seguridad `/verify` (`VerifyEmail.jsx`) que contiene la interfaz interactiva para códigos de un solo uso (OTP). Programó con `useRef` la lógica de salto automático de foco entre las 6 casillas numéricas para una experiencia fluida.
  - Configuró la lógica de enrutamiento y estado post-registro: al validar el código OTP, el sistema detecta vía parámetros de URL el tipo de cuenta. Si es ciudadano, otorga éxito total; si es institución, detiene el acceso directo notificando que la cuenta entró a un estado "pendiente" para revisión administrativa municipal.

### 👨‍💻 Juan (~32 hs | SP: 5)
**Historias:** HU-09 (Gestión de Roles y Permisos).

* **Implementaciones realizadas:** 
  - Estructuró el esquema relacional de los datos que el frontend enviará a la Base de Datos MySQL, preparando el terreno para el backend. Centralizó la información de acceso en una única tabla de `usuarios` optimizada.
  - Sentó las bases lógicas para el control de acceso basado en roles (RBAC), definiendo restricciones estrictas mediante tipos `ENUM` para los proveedores de autenticación (`local`, `google`, `facebook`) y blindando los estados del trámite de las instituciones.
  - Diseñó teóricamente la capa de integración de seguridad para JSON Web Tokens (JWT). Diagramó la arquitectura de una tabla `refresh_tokens` vinculada por llaves foráneas, lo cual permitirá al frontend mantener sesiones persistentes sin comprometer al servidor.

### 👩‍💻 Ludmila (~32 hs | SP: 9)
**Historias:** HU-07 (Inicio de sesión), HU-08 (Recuperación de contraseña), HU-10 (Edición de perfil), HU-11 (Eliminación de cuenta).

* **Implementaciones realizadas:** 
  - Construyó el módulo completo de autenticación (`Login.jsx`), integrando interfaces limpias para el inicio de sesión tradicional y botones modulares para *Social Login* (Google/Facebook).
  - Maquetó el flujo de recuperación de contraseña, asegurando consistencia visual a través del *layout* base del proyecto.
  - Aplicó reestructuraciones importantes al CSS global. Estandarizó el comportamiento del DOM asegurando que, gracias a utilidades y propiedades flexbox avanzadas, los formularios se mantengan perfectamente centrados en la sección activa de la pantalla compartida, sin importar su longitud.
  - Dejó preparada la arquitectura de navegación y los contenedores (*placeholders*) para el panel interno del usuario, sentando el puente para las futuras historias de edición y eliminación de perfil.

## 3. Conclusión del Sprint

Concluimos el Sprint 1 logrando un módulo de registro y autenticación frontend altamente escalable y pulido. Hemos establecido un estándar de código limpio mediante la separación de la lógica de estado y la interfaz visual, implementado un control de formularios eficiente con librerías modernas, y forjado una arquitectura de componentes en React lista para ser conectada a las APIs del servidor en la próxima iteración.
