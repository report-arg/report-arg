# Resumen de Migración Frontend y Backend - ReportARG

Este documento detalla exhaustivamente todo el trabajo realizado en la plataforma durante esta fase de desarrollo, abarcando desde la reestructuración de la interfaz de usuario hasta la preparación del backend y base de datos para los nuevos requerimientos de seguridad y geolocalización.

---

## 1. Estado Original (Antes de la intervención)

Antes de iniciar con las modificaciones, el proyecto ya contaba con una estructura funcional basada en Next.js (App Router). El estado base era el siguiente:

- **Estructura de Carpetas:** Estaba implementada la organización por rutas en `src/app` y componentes en `src/components`.
- **Formularios de Registro:** Existían formularios de registro monolíticos de un solo paso tanto para Ciudadanos como para Instituciones, con un esquema de base de datos que agrupaba "Nombre y Apellido" en un solo campo y no requería datos demográficos.
- **Validaciones:** Se utilizaba `react-hook-form` y `zod` para validaciones, pero carecían de comprobación de pasos intermedios.
- **Autenticación (Teórica):** Había una intención de usar JWT, pero la interfaz carecía del flujo necesario para la verificación de correos y cuentas inactivas.
- **Estilos Generales:** Existían los archivos `login.css` y `globals.css` configurados con las variables de color corporativo (`--azul`, etc.).

---

## 2. Modificaciones al Frontend (Paso a Paso)

### 2.1 Refactorización de Formularios a "Multi-Step Wizard"
Transformamos los formularios interminables de una sola página a un asistente en pasos, mejorando radicalmente la Experiencia de Usuario (UX) y reduciendo el abandono del registro.

- **Creación de `StepIndicator.jsx`:** Un componente visual dinámico que muestra al usuario su progreso: `(1) Cuenta -> (2) Datos -> (3) Ubicación`.
- **División de Datos (Ciudadanos):** 
  - Se dividió el campo genérico "Nombre Completo" a campos estrictos de `nombre` y `apellido`.
  - Se implementó un *trigger* asíncrono para validar los datos personales antes de permitir avanzar a la siguiente pestaña.
- **División de Datos (Instituciones):**
  - Se reorganizó el formulario para pedir la "Dirección exacta" recién en la última pestaña (Ubicación).

### 2.2 Nuevo Componente de Ubicación Geográfica
Se agregó el **Paso 3** estandarizado tanto para ciudadanos como instituciones, con el fin de descentralizar la plataforma hacia otras municipalidades en un futuro.

- **Campos Dropdown:** Se reemplazaron inputs de texto sueltos por etiquetas `<select>` para `provincia`, `ciudad`, y `zona`.
- **Restricción de Validaciones (Zod):** 
  - Se agregaron las opciones en blanco predeterminadas (`value=""`) para forzar que el usuario haga clic obligatoriamente y elija las opciones válidas: "Entre Ríos", "Viale", y "No Aplica".
  - El campo `zona` dejó de ser opcional en código (`z.string().optional()`) a ser de selección obligatoria (`z.string().min(1)`).

### 2.3 Refactorización Modular del Código (Clean Code)
Para evitar archivos de más de 300 líneas difíciles de mantener, se dividió la interfaz de usuario extraída de `CitizenRegister.jsx` e `InstitutionRegister.jsx` en componentes más pequeños:
- Creados en `src/components/auth/`:
  - `CitizenDataStep.jsx`
  - `CitizenLocationStep.jsx`
  - `InstitutionDataStep.jsx`
  - `InstitutionLocationStep.jsx`
- Ahora los archivos principales de registro actúan únicamente como un "Controlador" (almacenando el estado y la lógica de envío) mientras los nuevos subcomponentes se encargan puramente del diseño HTML/JSX.

### 2.4 Flujo de Verificación de Email (OTP)
Se corrigió la falla lógica donde la interfaz informaba "Cuenta Creada Exitosamente" antes de que el usuario verifique su correo.

- **Componente `VerifyEmail.jsx`:** Interfaz de 6 inputs (`cajitas`) con salto de foco automático. 
- **Lógica de Redirección Silenciosa:** Ahora, cuando el usuario termina el formulario, es enviado silenciosamente a la pantalla de verificación. Solo al ingresar el código OTP de 6 dígitos correcto aparece el mensaje de "Cuenta Creada Exitosamente" para Ciudadanos.
- **Estado Administrativo (Instituciones):** Si el usuario registrado era una Institución (detectado vía URL `?type=institution`), el mensaje de éxito cambia a *"¡Correo verificado! Tu solicitud fue enviada para revisión administrativa"*, informando que aún no pueden acceder.

### 2.5 Diseño y Estilos UI/UX
- **Centrado y Responsive:** Se corrigió un error crítico en `reset-password.css` agregando `margin: auto` para que el formulario de recuperación se mantenga centrado en la pantalla partida.
- **Selectores Premium:** Se aplicaron clases CSS personalizadas (`.form-select`) para estilizar los dropdowns nativos con una flecha minimalista y mejor espaciado.
- **Proporciones de Botones:** En las pantallas finales, el botón "Atrás" se redujo al 35% del espacio, mientras que "Verificar Correo" domina el 65%, guiando psicológicamente al usuario hacia la acción principal con transiciones de `hover` (.btn-secondary).

## 3. Modificaciones al Esquema de Base de Datos (MySQL)

Para que el backend pueda soportar el nuevo Frontend y la escalabilidad solicitada, se definió un script de migración SQL para la estructura de las tablas `usuarios` e `instituciones`.

1. **Campos de Verificación:**
   - `email_verified` (BOOLEAN)
   - `verification_code` (VARCHAR)
   - `verification_expires` (DATETIME)
2. **Campos para Social Login (OAuth 2.0):**
   - El inicio de sesión con Google o Facebook hace que la contraseña ya no sea un requisito universal (`MODIFY COLUMN password VARCHAR(255) NULL`).
   - Se introdujo `auth_provider` (`ENUM('local', 'google', 'facebook')`) para identificar el origen del registro.
   - Se añadió `provider_id` para vincular cuentas de redes sociales.
3. **Flujo de Instituciones:**
   - Para manejar el acceso bloqueado de las instituciones hasta que la municipalidad las valide, se agregó un campo `status` en la tabla `instituciones` (`ENUM('pending', 'approved', 'rejected')`).
4. **Seguridad (JWT Refresh Tokens):**
   - Se estructuró la tabla relacional `refresh_tokens` para almacenar tokens de larga duración, impidiendo que a los ciudadanos se les cierre la sesión en la app constantemente, manteniendo la máxima seguridad del estandar JWT.
